# Policy Templates

This directory contains `warranty.md` templates for common autonomous agent use cases.

Each template is pre-configured for a specific deployment context and uses the canonical Sigil Lex typed-block format, which is the only format accepted by the Sigil Lex policy engine built into sigil-sign.

## Available Templates

| Template | Use Case |
|---|---|
| `venture-warranty.md` | Agentic venture capital deployment |
| `treasury-warranty.md` | Autonomous treasury management |
| `defi-warranty.md` | DeFi strategy execution |
| `yield-farming-warranty.md` | Yield farming and LP management |
| `dao-governance-warranty.md` | DAO treasury and governance actions |
| `corporate-governance-warranty.md` | Enterprise and corporate agent deployment |

## How to Use

1. Copy the template that matches your deployment context.
2. Open [Sigil Warrant](https://sigilcore.com) to complete and sign the policy.
3. The drafter generates an Ed25519 keypair in your browser, signs the canonical-JSON policy hash, and produces a signed `warranty.md` ready for deployment.
4. Configure your Sigil Lex enforcement service with your operator public key (`LEX_OPERATOR_PUBLIC_KEY`) so the runtime can verify the signature on every policy load.

## Format Reference

`warranty.md` is parsed as typed section blocks. A valid policy contains a root-level `version: <semver>` line and at least one of `## evm`, `## tool_calls`, or `## custom`. Unknown fields inside known blocks are rejected at parse time.

### `## evm` — EVM transaction enforcement

Hard limits on EVM intents. Violations return `DENIED` immediately.

- `max_transaction_eth` — per-transaction ceiling.
- `allowed_actions` — list of permitted action strings (e.g. `wallet.transfer`, `contract.call`, `token.approve`, `erc4337.userop`).
- `allowed_chains` — list of permitted chain IDs.
- `chain_actions` — optional per-chain action overrides; each entry is `"<chainId>": <action list>`.
- `consensus_threshold_eth` — optional. Amounts above this trigger a `PENDING` decision and a durable hold in Sigil Command awaiting human approval.
- `consensus_require_hold` — optional boolean. When `true`, the threshold cannot be bypassed; when `false`, it is informational.

### `## tool_calls` — Non-EVM agent tool enforcement

Hard limits on non-EVM tool invocations. Supported call types: `bash`, `web_fetch`, `file_write`, `wallet_sign`, `email.send`.

- `allowed` — list of permitted tool calls.
- `bash.blocked_commands` — optional command substrings to block.
- `web_fetch.blocked_domains` — optional domain blocklist.
- `file_write.blocked_paths` — optional path blocklist.
- `email.require_approval` — optional boolean to gate `email.send` behind a hold.

### `## custom` — Operator-defined deny rules

Evaluated against the full intent payload before EVM and tool-call rules. Any matching rule returns `DENIED` with the rule name in `violatedRule`.

- `deny_if.<intent_field> <operator> <value>` — operators: `contains`, `starts_with`, `ends_with`, `matches` (regex), `equals`, `not_equals`.
- `deny_string: "<literal>"` — blocks any intent whose string fields contain the literal.

### `## soft_limits` — Informational daily caps

Optional. Daily aggregate caps tracked across all transactions. Soft limits are informational; enforcement of stateful daily tracking is implementation-dependent.

- `daily_evm_limit_eth` — daily aggregate ceiling for EVM transactions.
- `daily_tool_calls` — daily aggregate ceiling for tool invocations.

### `## signature` — Operator signature

Generated by the Sigil Warrant signing tool. **Do not deploy an unsigned policy.**

- `sigil-sig: <base64url>` — Ed25519 signature over all content above the `## signature` line. Verified at runtime against `LEX_OPERATOR_PUBLIC_KEY`. Re-run the signing tool after any change to the policy content.
